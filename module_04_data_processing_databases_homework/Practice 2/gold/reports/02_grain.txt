1. ФОРМУЛИРОВКА GRAIN
--------------------------------------------------------------------------------

  Одна строка в таблице фактов (fact_sales) представляет собой:

    ОДНУ ПРОДАЖУ ОДНОГО ТОВАРА (line item) — атомарное событие продажи,
    идентифицируемое уникальным sales_id.

  Полное описание одной строки:

    Продажа quantity единиц product_id клиенту customer_id,
    оформленная сотрудником employee_id в магазине shop_id (город city_id)
    в момент времени sales_timestamp в рамках транзакции transaction_number.

  Естественный ключ зерна: sales_id


2. УРОВЕНЬ АГРЕГАЦИИ
--------------------------------------------------------------------------------

  Уровень: АТОМАРНЫЙ (транзакционный), без предварительной агрегации.

  Данные НЕ агрегируются при загрузке в fact-таблицу.
  Каждая строка silver.sales -> одна строка fact_sales.

  Агрегация выполняется только на этапе аналитических запросов (GROUP BY):

    Строка fact     ->  Агрегация в запросах
    ---------------     ---------------------
    1 продажа             SUM по дню / месяцу / году
    1 товар               SUM по категории / магазину / стране
    1 транзакция          COUNT / AVG в разрезе измерений

  Примечание по данным EcoMarket:
    В текущем датасете transaction_number уникален для каждой строки
    (1 товар = 1 чек). Зерно line item совпадает с зерном транзакции,
    но в модели зафиксировано именно line item — как наиболее детальный
    и корректный уровень для розничных продаж.


3. ОБОСНОВАНИЕ ВЫБОРА GRAIN
--------------------------------------------------------------------------------

  3.1. Соответствие источнику данных
    - silver.sales уже хранит данные на уровне отдельной продажи (sales_id)
    - метрики quantity, discount, total_price относятся к одной позиции,
      а не к чеку или периоду

  3.2. Гибкость аналитики
    - из атомарного зерна можно агрегировать на любой уровень:
      транзакция, день, месяц, магазин, категория, сотрудник
    - невозможно корректно «развернуть» агрегат обратно в детали

  3.3. Отсутствие двойного счёта
    - total_price каждой строки учитывается ровно один раз
    - при SUM(total_price) не происходит повторного подсчёта одной продажи

  3.4. Соответствие бизнес-процессу (Этап 1)
    - ключевое событие — продажа товара, а не «итог за день» или «сводка
      по магазину»
    - все бизнес-вопросы решаются агрегацией из единого зерна


4. СООТВЕТСТВИЕ FOREIGN KEYS ЗЕРНУ
--------------------------------------------------------------------------------

  Правило: каждый FK в fact-таблице — связь «многие к одному» (N:1).
  Одна строка fact ссылается ровно на ОДНУ запись каждого измерения.

  FK в fact_sales          | Измерение        | Связь   | Соответствие grain
  -------------------------|------------------|---------|---------------------------
  date_key                 | dim_date         | N : 1   | один момент -> один день
  product_key              | dim_product      | N : 1   | один товар на строку
  customer_key             | dim_customer     | N : 1   | один клиент на строку
  employee_key             | dim_employee     | N : 1   | один сотрудник на строку
  shop_key                 | dim_shop         | N : 1   | один магазин на строку
  city_key                 | dim_city         | N : 1   | один город на строку

  Иерархии (страна, категория) — через измерения, НЕ через fact:

    dim_city     -> dim_country   (город определяет страну)
    dim_product  -> dim_category  (товар определяет категорию)

  Так избегается связь «многие ко многим» внутри fact-таблицы.


5. ЧТО НЕЛЬЗЯ ДЕЛАТЬ (ANTI-PATTERNS)
--------------------------------------------------------------------------------

  [X] Смешивать line items и итоги по транзакции в одной fact-таблице
  [X] Добавлять строки с агрегатами «Итого за месяц» / «Итого по магазину»
  [X] Хранить category_key и product_key как независимые FK, если категория
      полностью определяется товаром (достаточно product_key)
  [X] Дублировать country_key в fact, если city_key уже задаёт географию
  [X] Хранить в fact вычисленные агрегаты (monthly_revenue, avg_check и т.п.)

  Последствия нарушения: двойной счёт, некорректные SUM/COUNT, невозможность
  однозначно интерпретировать одну строку fact.


6. МЕТРИКИ И ИХ УРОВЕНЬ
--------------------------------------------------------------------------------

  Все метрики fact_sales относятся к ОДНОМУ зерну (одной продаже):

  Метрика       | Поле         | Уровень детализации
  --------------|--------------|------------------------------------------
  quantity      | quantity     | кол-во единиц в данной продаже
  discount      | discount     | скидка на данную позицию
  total_price   | total_price  | выручка по данной позиции

  Допустимые degenerate dimensions (хранятся в fact, не выносятся в dim):

    sales_id             — уникальный идентификатор зерна
    transaction_number   — номер чека (контекст транзакции)


7. ВЫВОД
--------------------------------------------------------------------------------

  Grain:     одна строка fact_sales = одна продажа одного товара (sales_id)

  Агрегация: атомарный уровень; группировка — только в аналитических запросах

  Обоснование: соответствие silver.sales, гибкость аналитики,
               отсутствие двойного счёта, корректные N:1 связи с измерениями

  FK:        date, product, customer, employee, shop, city — строго N:1
